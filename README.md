# Leyo

Leyo is a type-safe observable permission-based bytecode language and runtime, written in C. The repository includes the lexer, parser, bytecode packer, disassembler, and VM that power the language end to end.

We keep the project intentionally small and traceable:

- compiler and VM changes stay in sync
- bytecode is easy to inspect
- logs are treated as first-class debugging output
- the CLI stays simple enough for contributors to reason about quickly

## Quick Start

Build the project:

```bash
make
```

Build a source file:

```bash
leyo build test.leyo
```

Run the smoke test:

```bash
leyo test
```

Run generated bytecode:

```bash
leyo run a.lybc
```

Disassemble bytecode:

```bash
leyo disassemble a.lybc
```

## Current CLI

The executable currently supports these commands:

- `build <source> [output]`
- `run <file.lybc>`
- `disassemble <file.lybc> [--hex] [--head]`
- `help [query]`

Examples:

```bash
./bin/leyo.exe build test.leyo out.lybc
./bin/leyo.exe run out.lybc
./bin/leyo.exe disassemble out.lybc --hex
./bin/leyo.exe disassemble out.lybc --head
./bin/leyo.exe help build
```

`build` defaults the output file to `a.lybc` when you do not pass a destination.

Logging is configured in [`.lyst`](.lyst):

- `logging/enabled` turns file logging on or off
- `logging/build`, `logging/runtime`, `logging/controller`, and `logging/errors` toggle categories
- `logging/path` chooses the active log file
- `logging/retention_days` controls how old logs must be before cleanup runs
- `logging/retention_action` chooses `archive` or `delete`
- `logging/archive_path` chooses where archived logs are stored
- old active logs are still rotated into timestamped `.lylog` files in the log directory

Use `leyo init` to open the setup wizard, or `leyo init --defaults` to write the default `.lyst` immediately.

## Future CLI Ideas

These are the next commands and flags I would like us to grow into:

- `help` to print a real command summary
- `version` to print version/build metadata
- `log <file>` to change the active log destination
- `repl` for interactive experiments
- `test` for built-in language regression checks
- `build --debug` to keep extra compiler diagnostics visible
- `build --emit-bytecode` to dump raw bytecode artifacts alongside the wrapped file
- `run --trace` to step through VM execution with verbose opcode output
- `disassemble --json` for tooling-friendly output
- `fmt` for source formatting once the syntax settles
- `check` for compile-only validation without writing output

## Project Layout

- `src/` main implementation
- `include/` public headers shared across modules
- `docs/` contributor-facing notes and reference docs
- `archive/` older experiments and reference implementations
- `bin/` compiled executable output
- `build/` intermediate object files
- `logs/` runtime and compiler logs
