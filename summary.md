# Leyo Project Summary

## What This Project Is

Leyo is a small bytecode language and runtime written in C. The repository contains the full pipeline needed to go from source text to executable bytecode:

1. Lex source code into tokens
2. Parse tokens into bytecode
3. Wrap bytecode in a Leyo file header
4. Load and execute the file in a virtual machine
5. Disassemble bytecode for debugging

The project is intentionally compact so that compiler changes and VM changes stay easy to reason about together.

## High-Level Goal

The main goal of the project is to grow a simple but traceable language implementation that is easy to debug, easy to inspect, and easy for contributors or other agents to extend without having to learn a large framework first.

## Current User-Facing Commands

The CLI lives in [`src/main.c`](src/main.c) and currently supports:

- `build <source> [output]`
- `run <file.lybc>`
- `disassemble <file.lybc> [--hex] [--head]`
- `repl` exists, but it currently reports that the REPL is not built
- `test` is present in the command switch, but not implemented yet

The default build output is `a.lybc` when no output file is provided.

## End-To-End Flow

### Build path

The build command currently does this:

1. Reads a `.leyo` source file from disk
2. Tokenizes it with the lexer in [`src/lexer.c`](src/lexer.c)
3. Parses it into bytecode in [`src/parser.c`](src/parser.c)
4. Prepends a Leyo header in [`src/headerer.c`](src/headerer.c)
5. Writes the final `.lybc` file

### Run path

The run command does this:

1. Opens a `.lybc` file
2. Reads and validates the header
3. Loads the bytecode section
4. Loads the serialized constant pool if present
5. Executes the program in the VM in [`src/vm.c`](src/vm.c)

### Disassembly path

The disassembler in [`src/disassembler.c`](src/disassembler.c) prints either:

- a human-readable opcode listing
- raw hex bytes with `--hex`
- bytecode only, skipping the file header with `--head`

## Project Structure

- [`src/`](src) contains the main implementation
- [`include/`](include) contains shared headers and public structures
- [`docs/`](docs) contains design notes and reference documentation
- [`scripts/`](scripts) contains smoke-test helpers
- [`archive/`](archive) contains older experiments and reference code
- [`bin/`](bin) contains built executables
- [`build/`](build) contains intermediate build output
- [`logs/`](logs) contains runtime and compiler logs

## Main Modules

### Lexer

[`src/lexer.c`](src/lexer.c) turns source text into a token stream.

It currently understands:

- identifiers
- numbers
- strings
- chars
- operators
- semicolons
- braces and brackets
- commas
- condition tokens
- native-command tokens
- end-of-stream markers

Comments use `~` as the delimiter in the current implementation.

### Parser

[`src/parser.c`](src/parser.c) consumes tokens and emits bytecode.

The parser currently supports:

- variable declarations using the types `int`, `str`, `flt`, and `chr`
- assignments to previously declared globals
- simple binary expressions with `+`, `-`, `*`, `/`, and `^`
- a constant pool for literals

This is still a small language. Expression coverage is limited compared with a mature compiler.

### Bytecode and Headering

Opcode values live in [`include/bytecode.h`](include/bytecode.h).
The file wrapper lives in [`include/headerer.h`](include/headerer.h).

Current `.lybc` files contain:

1. A Leyo header
2. The instruction stream
3. The serialized constant pool

This means compiler and VM changes must stay in sync whenever file layout or opcode behavior changes.

### Virtual Machine

[`src/vm.c`](src/vm.c) implements the runtime.

The VM currently uses:

- registers `A`, `B`, and `R`
- a global variable table
- a speed stack
- a decoded constant pool
- an instruction pointer

The implemented opcodes cover register setup, arithmetic, global load/store, constant loading, a native-call hook, and program termination.

### Disassembler

[`src/disassembler.c`](src/disassembler.c) is a small debugging aid for viewing opcode streams.

It is useful when:

- a build looks wrong
- you want to inspect emitted bytecode before running it
- you need to compare parser output against VM behavior

## Important Data Structures

The most useful public types live in these headers:

- [`include/type.h`](include/type.h) defines token kinds and token streams
- [`include/parser.h`](include/parser.h) defines bytecode result structures, constant values, and global slots
- [`include/bytecode.h`](include/bytecode.h) defines the opcode set
- [`include/headerer.h`](include/headerer.h) defines the file header layout

## Testing And Validation

The main smoke test is [`scripts/test.ps1`](scripts/test.ps1).

It does three things:

1. Builds a sample source file
2. Runs the generated bytecode
3. Disassembles the output

This is the best quick check when changing the lexer, parser, header format, or VM.

## Current Limitations

- The REPL is present in the CLI but not actually built out
- The parser is still limited and does not cover a full language yet
- Bytecode compatibility is fragile because parser, header, disassembler, and VM all depend on the same layout
- Some command-line handling is still rough around the edges
- The project is still in an early evolution stage, so older docs and archive code may not match the newest behavior exactly

## Practical Notes For Contributors Or Agents

- When changing bytecode, update the parser, VM, and disassembler together
- When changing file layout, update the header writer and the VM loader together
- Check `logs/latest.lylog` when debugging failed runs
- Prefer the current code over older notes in `archive/` if they disagree
- Use the smoke test before and after making behavior changes

## Short Version

Leyo is a compact C-based bytecode language with a lexer, parser, bytecode wrapper, disassembler, and VM. The codebase is small enough that most features touch several modules at once, so the safest workflow is to inspect the current source, keep compiler and runtime aligned, and verify changes with the smoke test.
